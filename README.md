## ◈ Featured Projects

<details>
<summary><b>⬡ CronaFlow — Resilient Distributed Task Scheduler</b></summary>
<br/>

A production-grade, distributed task scheduling engine built for high-throughput, fault-tolerant workloads. Designed with enterprise observability and horizontal scalability as first-class concerns.

<div align="center">

| Attribute | Detail |
|:---|:---|
| **Stack** | Spring Boot · Spring WebFlux · React · Redis Streams · MongoDB · JWT · Docker |
| **Scale** | 500+ tasks/sec stable ingestion throughput · sub-1.3s average scheduling latency |
| **Fault Tolerance** | Active-passive leader election · Redis distributed locks · Dead Letter Queue (DLQ) |
| **Pipeline Design** | DAG-based dependency execution · automatic orphaned task recovery · configurable retries |
| **Integrations** | Automated SMTP dispatch · Gemini API processing · GraphQL LeetCode scraper |
| **Security** | Custom JWT authentication · role-based access on live queue monitoring dashboard |
| **Repository** | [![GitHub](https://img.shields.io/badge/View%20on%20GitHub-7c3aed?style=flat-square&logo=github&logoColor=white)](https://github.com/kshitij3103) |

</div>

CronaFlow addresses one of the hardest problems in distributed systems: ensuring exactly-once task delivery at scale without a centralized bottleneck. The concurrency model uses Redis distributed locking with active-passive leader election so that even under horizontal pod scaling, duplicate execution is provably eliminated. Fault-tolerant DAG execution with a Dead Letter Queue ensures permanent failures are isolated — not silently dropped or infinitely retried.

<br/>
</details>

This project demonstrates real-world microservice design patterns — not just service decomposition, but operationally sound decisions: decoupled async communication via RabbitMQ eliminates tight inter-service coupling, Keycloak provides production-grade SSO without rolling custom auth, and Gemini API integration is cleanly isolated at a dedicated recommendation service boundary.

</details>

<summary><b>⬡ ALS Disease Progression Prediction — Clinical ML Pipeline</b></summary>
<br/>

An end-to-end machine learning research pipeline for forecasting ALS (Amyotrophic Lateral Sclerosis) progression rates, combining a neuro-fuzzy ANFIS ensemble with rigorous statistical benchmarking against classical ML baselines.

<div align="center">

| Attribute | Detail |
|:---|:---|
| **Stack** | Python · PyTorch · Scikit-Learn · XGBoost · Pandas · Seaborn |
| **Model** | 10-cluster ANFIS ensemble · K-Means clustering · backpropagation · Least Squares Estimation |
| **Feature Engineering** | 6 clinical domains · dynamic IQR outlier removal · fold-specific median imputation |
| **Benchmarking** | 12 model combinations · tournament-style evaluation vs XGBoost, Random Forest, Linear baselines |
| **Performance** | Pearson correlation ~0.42 · RMSD ~0.53 on held-out longitudinal test set |
| **Impact** | Automated clinical rule tables · feature importance charts · reproducible evaluation framework |
| **Repository** | [![GitHub](https://img.shields.io/badge/View%20on%20GitHub-7c3aed?style=flat-square&logo=github&logoColor=white)](https://github.com/kshitij3103) |

</div>

The core contribution is a rigorous, leak-proof evaluation framework: fold-specific imputation ensures no target leakage during cross-validation, and the tournament-style benchmark with RFE and Pearson feature selection provides statistically defensible model comparisons. This is clinical ML done correctly — not just a Jupyter notebook, but a reproducible, documented research pipeline.
